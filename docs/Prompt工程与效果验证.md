# Prompt 工程与效果验证

## 一、设计理念

本项目的 Prompt 工程遵循以下原则：

1. **结构化输出优先**：所有需要后处理的 Prompt 都要求 JSON 格式输出，减少解析歧义
2. **角色设定明确**：每个 Prompt 开头设定专业角色（HR 助手、标签生成助手等），约束输出范围
3. **格式强约束**：明确禁止额外解释、标题、格式标记，降低后处理成本
4. **优雅降级**：每个 AI 功能都有对应的 Mock 实现，LLM 失败时自动回退

---

## 二、双模架构

```
用户请求
    ↓
AIService 判断 LLM_PROVIDER
    ├── "mock" → MockLLM（正则/模板/规则引擎）
    └── 其他   → OpenAI 兼容 API
                    ├── 成功 → 返回结果
                    └── 失败 → 回退到 MockLLM
```

所有 5 个 AI 功能都遵循同一模式：
1. 检查 `provider == "mock"`，是则直接调用 MockLLM
2. 否则加载 Prompt 模板，调用真实 LLM
3. 解析结果，若解析失败则回退 MockLLM

## 三、Prompt 模板分析

### 3.1 简历信息提取（resume_extraction.txt）

**目标**：从非结构化简历文本提取 11 个标准化字段

**设计策略**：
- 提供完整的 JSON Schema 作为输出模板，消除格式歧义
- 明确缺失字段的默认值规则（空字符串或 0）
- skills 字段指定为数组格式

**Mock 实现**：9 组正则表达式逐字段提取（手机号 `1[3-9]\d{9}`、邮箱标准正则、37 个技术关键词匹配等）

### 3.2 候选人摘要（candidate_summary.txt）

**目标**：生成 2-3 句 HR 专业摘要

**设计策略**：
- 明确优先级：教育背景 > 核心技能 > 岗位匹配度
- 亮点优先原则：名校、丰富经验、稀缺技能应首先提及
- 长度约束：2-3 句话，适配 HR 快速阅读场景

**Mock 实现**：3 个预设模板随机选取，动态填充候选人字段

### 3.3 自动标签（auto_tagging.txt）

**目标**：生成 3-6 个分类标签

**设计策略**：
- 提供 5 大类标签参考体系，引导多维度打标
- 字数限制（每标签不超过 6 字），确保 UI 展示整洁
- JSON 数组输出格式

**Mock 实现**：规则引擎——985 校名单匹配、经验年限分级、技能关键词映射，最终截取前 6 个标签

### 3.4 跟进建议（followup_suggestion.txt）

**目标**：生成 1-2 句具体的跟进建议

**设计策略**：
- 行动导向：必须包含"下一步做什么"
- 紧急度感知：长时间未跟进自动提醒
- 个性化：结合候选人背景给建议

**Mock 实现**：7 阶段建议映射表 + 超 3 天未更新自动追加紧急提醒

### 3.5 Copilot 系统指令（copilot_system.txt）

**目标**：定义 HR Copilot 的人设和行为规则

**设计策略**：
- 5 项能力声明：数据查询、漏斗分析、跟进识别、流程问答、决策支持
- 4 条行为规则：中文回复、简洁实用、引用实时数据、坦诚告知不确定
- 上下文注入：每次对话自动追加看板统计 JSON

**Mock 实现**：5 组关键词意图检测（数量、跟进、漏斗、建议、问候），返回包含实时数据的预设回复

## 五、效果验证

### 验证方法

1. 使用 3 份样例简历（`sample_data/sample_resume_*.txt`）测试解析准确度
2. 对 14 条种子数据分别生成摘要、标签、跟进建议
3. 通过 Copilot 对话验证 5 种意图的响应质量

### 验证结论

- Mock 模式在演示场景下表现良好，所有功能可正常使用
- 简历解析准确率在结构清晰的简历上达到 90%+
- 标签生成覆盖主要分类维度，区分度足够
- Copilot 对话覆盖 HR 常见问题，响应自然专业
